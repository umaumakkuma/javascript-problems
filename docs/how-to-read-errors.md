# エラーメッセージの読み方ガイド

エラーは「怒られ」ではなく、**原因を特定するためのヒントの塊**です。
エラーを読んで原因を絞り込む力は、この問題集のゴールである「AIが書いたコードの正しさを自分で判断する力」の中核でもあります。AIに「動きません」とだけ伝える人と、「この行でTypeErrorが出ています」と伝えられる人では、得られる答えの質がまったく違います。

## まず見るのは2箇所だけ

エラーが出たら、長い出力に圧倒される前に次の2箇所だけ見てください。

```
/home/you/javascript-problems/basic/07-data-types-literals/problem1.js:25
    const typeofNumber = typeoff 42;
                                 ^^
SyntaxError: Unexpected number      ← ② エラーの種類とメッセージ
    at wrapSafe (node:internal/modules/cjs/loader:1638:18)
    at Module._compile (...)        ← ③ スタックトレース（最初は読み飛ばしてOK）
```

1. **① ファイル名と行番号**（1行目の末尾の `:25` など） — どのファイルの何行目で起きたか。まずその行を見に行く
2. **② エラーの種類とメッセージ** — 何が起きたか。下の表で意味を調べる

③のスタックトレース（`at ...`の羅列）は「そのエラーに至るまでの関数呼び出しの経路」です。慣れるまでは読み飛ばして構いません。読むときは**上の行ほどエラー発生地点に近い**とだけ覚えてください。

## エラーの種類と、この問題集でよく出る原因

| エラーの種類 | 意味 | この問題集でよくある原因 |
|---|---|---|
| `SyntaxError` | 文法として壊れていて、**実行前の解析段階**で門前払いされた | カッコや引用符の閉じ忘れ、`const`のスペルミス。**check.jsでは💥になる** |
| `ReferenceError` | その名前の変数・関数が存在しない | **未回答**（`xxx is not defined`は動作確認コードが回答の変数を見つけられない状態）、変数名のタイプミス |
| `TypeError` | 値の「型」に対してできない操作をした | `xxx is not a function`（メソッド名のタイプミス、そのNode.jsバージョンにないAPI）、`Cannot read properties of undefined`（存在しないプロパティを辿った） |
| `RangeError` | 値が許容範囲を超えた | 無限再帰（`Maximum call stack size exceeded`）など |

### 重要: SyntaxErrorだけは特別

`SyntaxError`以外のエラーは**実行中**に起きるので、動作確認コードのtry-catchが捕まえて「❌ エラー: ...」と表示してくれます。
`SyntaxError`だけは**実行が始まる前**（ファイル全体の解析時）に起きるため、try-catchでは捕まえられず、プロセスごとクラッシュします（check.jsの💥はこれです）。「ファイルの先頭のconsole.logすら表示されない」のがSyntaxErrorの特徴です。

## 原因を絞り込む手順

1. **エラーの種類とメッセージを読む**（上の表と照らす）
2. **表示された行番号の行をよく見る** — タイプミス、閉じ忘れがないか。その行に問題がなければ、**直前の数行**も見る（閉じ忘れは後ろの行のエラーとして出ることが多い）
3. **メッセージ中の名前に注目する** — `foo is not defined`なら、`foo`を宣言したつもりの場所と綴りを見比べる
4. それでも分からなければ、**問題文の【参考】にあるjsprimerの章**を読み直す
5. エラーメッセージでウェブ検索する（英語のままの方が情報が多い）

## 試してみよう

`basic/00-tutorial/problem0.js` の回答欄の行を一度削除して実行すると、安全に`ReferenceError`を体験できます。さらに、回答の行末の引用符を1つ消して実行すると`SyntaxError`（💥）も体験できます。「わざと壊して、エラーを読んで、直す」は最高の練習です。
